## 一、核心原则：数据主权归属

```
┌──────────────────────────────────────────────────┐
│  代码部署（开发者）                                  │
│  权力：✅ 改 UI/UX   ✅ 改数据结构   ✅ 首次种子数据    │
│  禁止：❌ 覆盖管理员已有数据                           │
├──────────────────────────────────────────────────┤
│  管理后台（管理员）                                  │
│  权力：✅ 增删改专家   ✅ 改分类颜色   ✅ 管理配置      │
│        ✅ 批量导入    ✅ 权限管理    ✅ 评分调整       │
│  数据终极所有权属于管理员                             │
└──────────────────────────────────────────────────┘
```

**localStorage 是唯一数据真理来源。`data.js` 仅在新用户首次访问时作为初始化种子，之后永远闭嘴。**

## 三、颜色锁定规则 🔒

**领域分类颜色仅由主管理员在后台「分类管理」中修改。**

```javascript
// 代码中共 3 处可写颜色的路径，全部受保护：

// 路径 1：v6 migration（仅对新领域或缺少颜色的领域设置默认值）
if (!f.color) f.color = src.color;  // ✅ 有守卫

// 路径 2：后台分类管理新增标签（管理员操作）
// 路径 3：后台分类管理修改颜色（管理员操作）
```

**三重锁机制**：
1. `if (!f.color)` 守卫 — 已有颜色绝不覆盖
2. 迁移注释块声明最高优先级
3. `MEMORY.md` 永久记录铁律

**常见陷阱**：沙盒直连 URL 是新 origin → localStorage 为空 → 加载 `data.js` 默认颜色。这不是覆盖，是新浏览器环境。管理员颜色仅存在于 `app.codebuddy.work` origin。

## 五、Migration 行为准则

### 允许的迁移（自动执行）
- 新增顶层属性（如 `permissions`、`sortOptions`、`uiConfig`）
- 数据结构转换（如 `contact` 单字段 → `contacts` 数组）
- 修复评分配置结构（`migrateRatingConfig`）
- 新增领域标签（`data.js` 有但 localStorage 没有的字段）
- 对**缺失**颜色的领域设置默认值（`if (!f.color)` 守卫）

### 禁止的迁移（自动执行）
- ❌ 从 `data.js` 读取值覆盖 localStorage 中已有的数据
- ❌ 覆盖管理员设置的领域颜色
- ❌ 恢复管理员故意清空的字段
- ❌ 修改管理员创建的专家信息

### 手动修复流
当需要批量修复数据时，由项目负责人下达明确指令，开发者编写一次性脚本或直接操作 localStorage 执行。

## 六、版本管理

### 两层版本号

| 变量 | 位置 | 作用 |
|------|------|------|
| `CURRENT_DB_VERSION` | `js/app.js` | 触发数据库 migration |
| `_latestVersion` | `js/app.js` | CDN 缓存刷新 + 蓝色更新提示条 |
| `?v=N` 参数 | `index.html` | 强制 CDN 拉取新文件 |

### 部署检查清单
1. 升级 `CURRENT_DB_VERSION`（如有数据结构变更）
2. 升级 `_latestVersion`
3. 更新 `index.html` 中所有 `?v=` 参数
4. 更新 `console.log` 版本号
5. 用 `app.codebuddy.work` 链接确认加载

## 八、常见问题

### Q: 为什么我部署后看不到管理员设置的颜色？
A: 你可能用了沙盒直连 URL。管理员颜色存在 `app.codebuddy.work` 的 localStorage 中。使用永久分享链接访问。

### Q: 管理员的数据会在我部署后被覆盖吗？
A: v3.3 起不会。代码迁移只添加新字段/结构，不覆盖已有数据值。

### Q: 如果我需要改 data.js 中的专家数据怎么办？
A: 新增专家：在 data.js 添加 → 部署后，已有用户通过管理后台「数据源同步」手动触发导入。  
修改已有专家：告知项目负责人 → 人工编写修复脚本 → 通过浏览器 console 或管理后台执行。
